4 · Flow thanh toán Stripe

3 giai đoạn: tạo PaymentIntent (server) → nhập thẻ (client ↔ Stripe, không qua server của bạn) → webhook (nguồn sự thật duy nhất).

⚠ Bug tiềm ẩn: trong orderService.go:155, webhook handler gọi s.sqsClient.PublishOrderEvent(...) không check nil (PlaceOrder thì có check). Nếu bật Stripe mà không cấu hình SQS → webhook đầu tiên sẽ panic.

5 · AWS SQS — vòng đời một message

SQS decouple API khỏi các tác vụ phụ trợ: API chỉ mất ~10ms để ném event rồi trả response; email/analytics/shipping xử lý sau, chậm hay chết cũng không ảnh hưởng việc đặt hàng. Click từng giai đoạn:

Mặc định: xem code producer của app này.

2 tính chất bắt buộc phải nhớ khi viết consumer

① At-least-once delivery — message có thể được giao ≥ 1 lần → consumer phải idempotent (xử lý 2 lần cùng order_id không được gửi 2 email).
② Standard queue không đảm bảo thứ tự — ORDER_PAID có thể đến trước ORDER_PLACED.

Setup SQS queue (kèm Dead Letter Queue)

# 1. Tạo DLQ trước
aws sqs create-queue --queue-name order-events-dlq

# 2. Lấy ARN của DLQ
aws sqs get-queue-attributes \
  --queue-url https://sqs.ap-southeast-1.amazonaws.com/123456789012/order-events-dlq \
  --attribute-names QueueArn

# 3. Tạo queue chính, lỗi 5 lần thì đẩy sang DLQ
aws sqs create-queue --queue-name order-events --attributes '{
  "VisibilityTimeout": "30",
  "RedrivePolicy": "{\"deadLetterTargetArn\":\"arn:aws:sqs:ap-southeast-1:123456789012:order-events-dlq\",\"maxReceiveCount\":\"5\"}"
}'

# Xem message trong queue (debug)
aws sqs receive-message --queue-url $QUEUE_URL --max-number-of-messages 10

Các điểm cải thiện đáng làm tiếp

① Bọc transaction quanh CreateOrder + ClearCart (hiện là 2 thao tác rời — app chết giữa chừng thì cart còn nguyên dù đơn đã tạo).
② Trừ tồn kho khi đặt hàng — PlaceOrder hiện không kiểm tra/giảm stock.
③ Lưu tiền dạng cents (int64) thay vì float64 — khớp với cách Stripe tính.
④ Chống replay webhook: check payment_status == paid trước khi publish lại ORDER_PAID lên SQS.
